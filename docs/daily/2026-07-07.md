# Diario - 2026-07-07

## Contexto

- Investigacao de falha de sincronizacao percebida em atividades apos 02/07/2026.

## Hipotese principal

- O webhook endurecido exige token no POST. Se a subscricao do Strava nao foi atualizada para usar o callback com `?token=...`, as novas atividades deixam de entrar automaticamente.
- Em paralelo, o sync incremental ficava vulneravel a uma atividade futura ou com data errada na base, porque o cursor usava cegamente a atividade mais recente salva.

## Mudancas

### 1. Clamp defensivo no cursor incremental

- **Arquivos:** `dash/app/api/strava/sync/route.ts`
- O sync incremental agora limita a data usada como cursor para no maximo `15 minutos` no futuro em relacao ao relogio do servidor.
- Isso evita que uma atividade com data errada ou futura bloqueie a subida de treinos reais mais recentes.
- O meta do usuario agora registra se houve clamp do cursor e guarda data original x efetiva para diagnostico.

### 2. Smoke tests do contrato de cursor

- **Arquivos:** `dash/scripts/test-smoke.mjs`, `dash/tests/security-smoke.test.mjs`
- Os testes de fumaca passaram a validar a existencia do clamp defensivo e do flag `latestSavedWasFutureClamped`.

## Validacao

- `npm run test:smoke` aprovado.
- `npm run typecheck` aprovado.


### 3. Plano pro reduzido para 3 anos

- **Arquivos:** `dash/lib/access.ts`, `dash/scripts/test-smoke.mjs`, `dash/tests/security-smoke.test.mjs`
- O limite do plano pago foi reduzido de `5 anos` para `3 anos`, mantendo `free = 2 anos` e `admin/master = acesso total`.
- Os smoke tests foram alinhados para validar o novo contrato comercial.

### 4. Cache anual em chunks para cortar leituras por tela

- **Arquivos:** `dash/lib/activity-cache.ts`, `dash/lib/firebase.ts`, `dash/app/api/activities/route.ts`, `dash/app/page.tsx`
- Foi criada uma camada de cache anual por usuario em `yearCaches/{ano}/chunks`, com indice por ano e blocos de atividades serializadas.
- O painel continua podendo carregar o recorte anual completo, mas a leitura saiu de centenas ou milhares de documentos de `activities` para poucos documentos de cache por ano.
- A home agora tenta reparar `availableYears` usando os indices de cache antes de cair em varredura completa da base.

### 5. Invalidacao de cache nos pontos que alteram a base

- **Arquivos:** `dash/app/api/strava/sync/route.ts`, `dash/app/api/strava/webhook/route.ts`, `dash/app/api/activities/[stravaId]/route.ts`, `dash/app/api/admin/backfill-efforts/route.ts`
- Sync incremental/full, webhook, exclusao logica de atividade e backfill de best efforts agora reconstruem apenas os anos afetados.
- O webhook deixou de fazer full scan da colecao inteira e passou a recalcular o meta a partir dos indices de cache anual.
- Isso desloca custo de leitura para eventos de mudanca de dados, em vez de repetir o mesmo custo em toda visita ao dashboard.

### 6. Historico paginado e cache local de anos

- **Arquivos:** `dash/app/api/activities/history/route.ts`, `dash/components/DashboardClient.tsx`, `dash/components/dashboard/types.ts`, `dash/app/page.tsx`
- O historico da tabela deixou de depender do array inteiro carregado no client e passou a buscar apenas a pagina ativa pela API.
- A nova rota aceita `years`, `sports`, `start`, `end`, `page` e `pageSize`, usando o cache anual do servidor para montar o recorte antes da paginacao.
- O client tambem passou a reutilizar anos ja baixados em `sessionStorage`, indexados por atleta e `lastSync`, reduzindo refetch e CPU em navegacao repetida.
